The results from Part IV-(a) show that the us-central1 server was consistently faster than the europe-west1 server. Both /register and /list averaged around 440 ms in us-central1, while europe-west1 was closer to 590–660 ms. This difference is most likely due to network distance. Since the client running the test was probably geographically closer to us-central1, the round-trip time to that region was lower. Requests to europe-west1 have to travel farther, which increases latency. As both instances use the same Firestore database, the main factor causing the difference is network delay rather than differences in server logic or database performance.

For the consistency experiment, 0 out of 200 registrations were missing when immediately checked from the other region. Even though Firestore is eventually consistent, replication across regions happens very quickly in practice. The delay is likely only a few milliseconds, which makes it hard to observe any inconsistency under light load. This insinuates that while the system doesn't guarantee strict immediate consistency, in normal conditions updates appear almost instantly across regions.
